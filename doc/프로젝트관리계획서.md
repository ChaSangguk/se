## 1. 서론

* **프로젝트 명** : 클라우드 파일 관리 시스템 구축
* **배경 및 목적**: 프로젝트의 요구사항 이행을 위한 지표설정
* **프로젝트 정의**: 조직 내 파일을 중앙 관리하고 팀원 및 외부 협력 업체와 안전하게 공유하는 웹 기반 시스템 구축

* **프로젝트 목표**:
    * 200명 이상의 직원이 동시 사용할 수 있는 안정적인 환경 제공.
    * 직관적인 업로드/다운로드 및 폴더 체계 제공.
    * 파일 버전 관리 및 복구 기능을 통한 데이터 안정성 확보

## 2. 개발 계획
* 사용 모델 : 초기 목표는 명확한 편이나,200명 이상의 다양한 사용자의 피드백이 다량 발생할 것으로 예상되므로, 사용자의 피드백을 빠르게 반영하기 위해 애자일 방법론 중 하나인 **스크럼** 모델을 채택
* **WBS** : 
    * 요구사항 분석
        * 사용자 요구사항 정의
        * 시스템 요구사항 정의
        * 요구사항 명세서 작성 및 검토
    * 설계
        * 시스템 아키텍처 및 DB 설계
        * API 설계
        * UI/UX 설계
        * 인프라 및 배포 구조 설계
    * 코딩
        * 백엔드 개발 (인증, 파일 관리, 버전 관리, 공유 기능)
        * 프론트엔드 개발 (대시보드, 파일 관리 UI, 사용자 관리)
        * 데이터베이스 구현
    * 테스트
        * 단위 및 통합 테스트
        * 성능 테스트 (200명 이상 동시 사용)
        * 보안 테스트
        * 사용자 인수 테스트 (UAT)

* 개발 일정 : 
    * **프로젝트 초기화 단계 (10일)**
        * 1-5일: 프로젝트 요약 및 팀 오리엔테이션
        * 6-10일: 프로젝트 계획 및 개발 환경 구축
    * **요구사항 분석 단계 (2주)**
        * 11-14일: 사용자/시스템 요구사항 수집 및 정의
        * 15-24일: 요구사항 분석 및 명세서 작성
    * **설계 단계 (2주)**
        * 25-31일: 시스템 아키텍처, DB, API 설계
        * 32-38일: UI/UX 설계 및 프로토타입 개발
    * **스프린트 반복 (20주, 2주 × 10회)**
        * 각 스프린트별 백엔드/프론트엔드 기능 구현 및 테스트
        * 매 스프린트마다 인수 테스트 및 피드백 반영
    * **최종 테스트 및 배포 (2주)**
        * 전체 통합 테스트, 성능/보안 테스트
        * 프로덕션 배포 및 모니터링

## 3. 팀 구성
* 팀 구조: 수평적 구조를 유지하되, 파일 관리와 보안이 중요하므로 역할 분담 명확화.
* 역할 분담:
    * PM: 전반적인 일정 및 산출물 검토.
    * Back-end: 스토리지 API, 버전 관리, DB 구축 담당. 총 3명
    * Front-end: 웹 기반 인터페이스 및 공유 기능 구현. 총 2명

## 4. 품질,산출물 관리
* 팀미팅 계획 :
    * 데일리 스크럼(매일 1회 20분간) 전인원 참석
    * 스프린트 리뷰 진행 (스프린트 종료 이후)
* 산출물 문서 규칙 : `클라우드_문서명_버전` 의 형식을 따름. 버전명의 경우에는 `x.y`의 형태를 따르며 사소한 내용 변경시 y의 값을 변경, 중대한 변경의 경우 `x`값을 하나 증가 후 `y`를 0으로 명명함
예시 : `클라우드_프로젝트관리계획서_2.12`
* 소프트웨어의 경우 `클라우드_버전`의 형식으로 저장 버전명은 산출물의 규칙과 동일하다
* 산출물 관리 : GitHub를 이용해 저장, 관리
    * git-flow 브랜치 방식을 이용하여 관리
    * 커밋 규칙 :
        * **커밋 메시지 형식**: `[Type] #이슈번호 - 간단한 설명`
        * **Type 종류**:
            * `[Feat]`: 새로운 기능 추가
            * `[Fix]`: 버그 수정
            * `[Refactor]`: 코드 리팩토링
            * `[Docs]`: 문서 작성 및 수정
            * `[Test]`: 테스트 코드 추가 및 수정
        * **예시**: `[Feat] #25 - 파일 업로드 기능 구현`, `[Fix] #42 - 로그인 버그 수정`
